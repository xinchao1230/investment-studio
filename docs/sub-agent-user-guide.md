## 1. What is a Sub-Agent

A Sub-Agent is a **specialized task agent** in OpenKosmos that is delegated specific tasks on demand by the main Agent (parent) during conversations. Unlike main Agents that interact directly with users, Sub-Agents autonomously complete tasks in the background and return results to the parent.

> **Analogy**: If the main Agent is a project manager, then Sub-Agents are domain experts — web researchers, code reviewers, documentation writers — each focusing on their area of expertise.

### Key Features

| Feature | Description |
|------|------|
| Pre-configured | Users create or install them in Settings beforehand; they are not dynamically generated by AI at runtime |
| Specialized | Each Sub-Agent has a defined role, toolset, and system prompt |
| Non-recursive | Sub-Agents cannot delegate tasks to other Sub-Agents (single layer only) |
| Model inheritance | Uniformly uses the parent Agent's LLM model |
| Isolated by default | Cannot access parent conversation context by default; can be configured to enable |
| Parallel support | The parent Agent can launch multiple Sub-Agents simultaneously for different tasks |

## 2. Core Concepts

### Two-Level Reference Model

Sub-Agents use the same **two-level reference model** as Skills:

```
Level 1: Global Registry (Managed in Settings)
  └─ Create or install all available Sub-Agents in Settings → Sub-Agents

Level 2: Agent-Level Reference (Selected in Agent Editor)
  └─ In a specific Agent's editor, check which Sub-Agents that Agent is allowed to use
```

Only Sub-Agents that are checked/referenced by an Agent will appear in that Agent's available tools list.

### Execution Flow

```
User sends message
    │
    ▼
Main Agent analyzes task ──→ Decides whether to delegate
    │                    │
    │              ┌─────┴─────┐
    │              ▼           ▼
    │       spawn_subagent  spawn_subagents
    │         (single task)    (multiple tasks in parallel)
    │              │           │
    │              ▼           ▼
    │        Sub-Agent executes autonomously (tool calls, reasoning loop)
    │              │           │
    │              ▼           ▼
    │        Returns results to main Agent
    │
    ▼
Main Agent integrates results and replies to user
```

## 3. Quick Start

Three steps to start using Sub-Agents:

### Step 1: Install a Sub-Agent

Go to **Settings → Sub-Agents → Add from Library** to browse and install suitable Sub-Agents from the OpenKosmos library.

### Step 2: Enable Sub-Agent for an Agent

Go to **Agent Editor → Sub-Agents Tab** and check the Sub-Agents you want to use.

### Step 3: Start a Conversation

Just have a normal conversation with the Agent. When a task matches a Sub-Agent's expertise, the Agent will automatically decide to delegate. For example:

> **User**: Research the latest developments in React Server Components, and review the code quality of `src/utils/api.ts` at the same time.
>
> **Agent**: Sure, I'll launch two tasks in parallel:
> - 🔍 **web-researcher** → Research React Server Components
> - 🔎 **code-reviewer** → Review api.ts
>
> *(Sub-Agent real-time execution progress is displayed in the chat...)*
>
> Combining the results from both Sub-Agents: ...

## 4. Managing Sub-Agents

### 4.1 Install from Library

1. Open **Settings → Sub-Agents**
2. Click the **"Add from Library"** button
3. Browse the Sub-Agent library; each entry shows:
   - Emoji icon and name
   - Functionality description
   - Required MCP servers and Skills
   - Context access mode
4. Click **"Install"** to install the desired Sub-Agent
5. After installation, it automatically appears in the Sub-Agents list

> 💡 **Tip**: OpenKosmos automatically checks for and updates installed library Sub-Agents on each startup.

### 4.2 Create Manually

1. Open **Settings → Sub-Agents**
2. Click the **"Create Custom"** button
3. Fill in the configuration form:
   - **Name**: Unique identifier (lowercase letters, numbers, and hyphens only, e.g., `web-researcher`)
   - **Display Name**: Display name (e.g., "Web Researcher")
   - **Emoji**: Icon (e.g., 🔍)
   - **Description**: Functionality description (the parent Agent uses this to decide when to delegate tasks)
   - **System Prompt**: Behavioral instructions (see [§10 Best Practices](#10-system-prompt-best-practices))
   - **Context Access**: Context access mode (see [§5.2](#52-context-access-modes))
   - **Max Turns**: Maximum conversation turns (default 25)
   - **MCP Servers / Skills / Knowledge Base**: Capability configuration (see [§5.3](#53-capability-inheritance-mechanism))
4. Click **"Save"** to complete creation

### 4.3 Edit a Sub-Agent

In the Sub-Agents list, click the **"⋯"** menu on an entry → **"Edit"** to enter the edit page and modify the configuration.

### 4.4 Delete a Sub-Agent

In the Sub-Agents list, click the **"⋯"** menu on an entry → **"Delete"** to confirm and delete.

> ⚠️ After deletion, all Agents referencing this Sub-Agent will be automatically disassociated.

## 5. Configuration Details

### 5.1 Basic Information

| Field | Required | Description |
|------|:---:|------|
| **Name** | ✅ | Unique identifier, format: lowercase letters + numbers + hyphens (e.g., `code-reviewer`) |
| **Display Name** | ✅ | UI display name (e.g., "Code Reviewer") |
| **Description** | ✅ | Functionality description — **very important**, the parent Agent relies on it to decide whether to delegate tasks |
| **Emoji** | ✅ | Icon, default 🤖 |
| **System Prompt** | ✅ | Defines the Sub-Agent's behavior, expertise, and workflow |
| **Context Access** | ✅ | Context access mode, default `isolated` |
| **Max Turns** | Optional | Maximum conversation turns, default 25, range 1–100 |
| **Workspace** | Optional | Independent workspace directory path |
| **Version** | Auto | Semantic version number |

### 5.2 Context Access Modes

The context access mode determines how much parent conversation information the Sub-Agent can access:

#### `isolated` (Isolated Mode, Default)

- Sub-Agent **has no access** to parent conversation history
- Only receives the task description
- **Use case**: Independent tasks — web search, code review, file processing
- **Advantage**: Lowest token consumption, best privacy, fastest execution

#### `parent_summary` (Summary Mode)

- Sub-Agent receives an **LLM-generated structured summary** of the parent conversation (~500–800 tokens)
- Summary includes: conversation overview, completed work, current issues, key resource paths, etc.
- **Use case**: Tasks that need conversation background but not complete details
- **Advantage**: Balances context awareness with token efficiency

#### `full_history` (Full History Mode)

- Sub-Agent receives the **complete parent conversation messages**
- **Use case**: Tasks requiring deep understanding of previous discussion content
- **Note**: ⚠️ Heavy token consumption, use only when truly needed

> 💡 **Recommendation**: Prefer `isolated`, then `parent_summary`, and use `full_history` only when necessary.

### 5.3 Capability Inheritance Mechanism

Sub-Agents **inherit the parent Agent's** MCP Servers, Skills, and Knowledge Base configuration by default, with support for independent configuration overrides.

#### MCP Server Inheritance

| `inherit_mcp_servers` | Behavior |
|:---:|------|
| `true` (default) | Sub-Agent's own MCP servers are **merged** with the parent's; same-name servers prioritize Sub-Agent's |
| `false` | Only uses MCP servers configured on the Sub-Agent itself |

#### Skills Inheritance

| `inherit_skills` | Behavior |
|:---:|------|
| `true` (default) | Sub-Agent's own Skills are **merged** with the parent's (deduplicated) |
| `false` | Only uses Skills configured on the Sub-Agent itself |

#### Knowledge Base Inheritance

| `inherit_knowledge_base` | Behavior |
|:---:|------|
| `true` (default) | If the Sub-Agent has no KB path configured, uses the parent's |
| `false` | Does not inherit; even if the Sub-Agent has no configuration, the parent's knowledge base is not used |

> 💡 **Common configurations**:
> - **Full inheritance**: All three set to `true` + Sub-Agent's own lists empty → fully uses the parent's capabilities
> - **Partial extension**: `inherit_mcp_servers = true` + additionally add Sub-Agent-specific MCP servers
> - **Fully independent**: All three set to `false` → Sub-Agent uses a completely independent toolset

## 6. Assigning Sub-Agents to an Agent

1. Select an Agent on the main interface and click to enter **Agent Settings**
2. Switch to the **"Sub-Agents"** Tab
3. **Check** the Sub-Agents you want to enable from the list
4. Each entry shows:
   - ✅ / ☐ Check status
   - Emoji + display name
   - Functionality description
   - Context access mode label
5. Changes are saved automatically when checked

You can also select **"Apply to Agents…"** from the **"⋯"** menu in the Sub-Agents list to batch-enable a Sub-Agent for multiple Agents via a dialog.

> 💡 **Tip**: After enabling a Sub-Agent, the main Agent's system prompt is automatically injected with an "Available Sub-Agents" section that tells the AI which sub-agents are available and how to use them.

## 7. Runtime Experience

### 7.1 Single Task Execution

When the main Agent decides to delegate a single task, it calls the `spawn_subagent` tool. A Sub-Agent execution card appears in the chat interface:

```
┌─────────────────────────────────────────────────┐
│ 🔍 Sub-Agent: web-researcher       ⏳ Turn 3/25 │
│ Task: Research latest React Server Components developments │
│ ┌─────────────────────────────────────────────┐  │
│ │ ✅ bing_web_search        "react server..."  │  │
│ │ ✅ fetch_web_content      "reactjs.org/..."  │  │
│ │ ⏳ fetch_web_content      running...          │  │
│ │ 💬 Found 3 related articles, organizing...    │  │
│ └─────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 7.2 Parallel Execution

When multiple tasks need to be executed simultaneously, the main Agent calls `spawn_subagents`, with up to **5 Sub-Agents running in parallel**:

```
┌─────────────────────────────────────────────────┐
│ 🤖 Parallel Sub-Agents (3 tasks)    ⏳ Running   │
│                                                   │
│ ┌─────────────────────────────────────────────┐  │
│ │ 🔍 web-researcher      ⏳ Turn 3/25         │  │
│ │ [✅ bing_web_search] [⏳ fetch_web_content]  │  │
│ ├─────────────────────────────────────────────┤  │
│ │ 🔎 code-reviewer       ✅ Done (12.1s)      │  │
│ │ [✅ read_file] [✅ search_file_contents]     │  │
│ ├─────────────────────────────────────────────┤  │
│ │ 📝 doc-writer           ⏳ Turn 1/25         │  │
│ │ (no steps yet)                               │  │
│ └─────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 7.3 Real-Time Progress Display

Real-time status updates during execution include:

| Information | Description |
|------|------|
| **Turn progress** | Current turn / max turns (e.g., "Turn 4/25") |
| **Tool call log** | Each tool's name + execution status (✅ done / ⏳ running / ❌ error) |
| **Tool argument summary** | Preview of key parameters for tool calls |
| **Text summary** | Summary of the most recent LLM text output |
| **Final status** | ✅ Done (success) / ❌ Failed (failure) + execution duration |

After the Sub-Agent completes, results are automatically returned to the main Agent, which integrates them into the reply.

## 8. AGENT.md File Format

Sub-Agents are stored as `AGENT.md` files using YAML front-matter + Markdown body format, compatible with Claude Code's Sub-Agent specification.

### 8.1 Format Specification

```
---
[YAML Front-matter: configuration fields]
---

[Markdown Body: System Prompt content]
```

**Rules**:
1. The file must start with `---`; the YAML front-matter ends with the second `---`
2. `name` and `description` are required fields
3. The Markdown body section serves as the `system_prompt`
4. OpenKosmos extension fields are placed under the `x-kosmos` namespace
5. Unknown fields are silently ignored during import (forward compatible)

### 8.2 Complete Example

```markdown
---
# === Claude Code Standard Fields ===
name: web-researcher
description: >-
  Specialized web research agent. Delegate tasks involving
  internet research, news gathering, and information synthesis.
model: inherit
maxTurns: 20
tools:
  - WebSearch
  - WebFetch
mcpServers: []
skills: []

# === OpenKosmos Extension Fields ===
x-kosmos:
  display_name: Web Researcher
  emoji: "🔍"
  version: "1.0.0"
  context_access: parent_summary
  builtin_tools:
    - bing_web_search
    - google_web_search
    - fetch_web_content
    - read_html_by_url
  inherit_mcp_servers: true
  inherit_skills: false
  inherit_knowledge_base: false
---

You are a specialized web research agent. Your task is to find,
analyze, and synthesize information from the internet.

## Workflow

1. **Understand the query**: Parse the research topic and identify key
   search terms
2. **Search broadly**: Use web search tools to find relevant sources
3. **Deep dive**: Fetch and read the most promising pages
4. **Synthesize**: Compile findings into a clear, structured report

## Output Format

Provide a structured research report including:
- **Summary**: 2-3 sentence overview
- **Key Findings**: Bullet-pointed main discoveries
- **Sources**: URLs with brief descriptions
- **Recommendations**: Actionable next steps (if applicable)

## Guidelines

- Prioritize authoritative and recent sources
- Cross-reference facts from multiple sources
- Note any conflicting information found
- Always include source URLs for verification
- Stay focused on the assigned research topic
```

### 8.3 Claude Code Compatibility

OpenKosmos's AGENT.md format is fully compatible with Claude Code's Sub-Agent definitions:

| Claude Code Field | OpenKosmos Equivalent | Compatibility Handling |
|-----------------|------------|---------|
| `name` | `name` | Direct mapping |
| `description` | `description` | Direct mapping |
| `model` | Ignored (always uses parent model) | `inherit` is already the default |
| `maxTurns` | `max_turns` | Automatic conversion |
| `tools` | Mapped to `builtin_tools` as reference | Best-effort mapping of Claude Code tool names to OpenKosmos built-in tools |
| `skills` | `skills` | Direct mapping |
| `mcpServers` | `mcp_servers` | Structure conversion |

**Importing pure Claude Code format**: AGENT.md files without `x-kosmos` can be imported normally; missing OpenKosmos fields will use defaults:

- `emoji` → 🤖
- `context_access` → `isolated`
- `version` → `1.0.0`
- Inheritance flags → all `true`

## 9. Sub-Agent Design Patterns

### 9.1 Tool Specialist

**Characteristics**: Focused on operating a specific toolset with clear responsibilities.

**Use case**: Code review, file processing, data analysis

**Typical configuration**:
```yaml
name: code-reviewer
description: Senior code reviewer for quality assurance.
x-kosmos:
  context_access: isolated
  builtin_tools:
    - read_file
    - search_file_contents
    - search_files
  inherit_mcp_servers: false
  inherit_skills: false
```

**System Prompt key points**:
- Define clear review criteria and checklist items
- Define output format (issue categorization, severity, fix suggestions)
- Restrict scope of operations (read-only, no file modifications)

---

### 9.2 Research Assistant

**Characteristics**: Excels at information retrieval and content synthesis.

**Use case**: Web research, document search, knowledge organization

**Typical configuration**:
```yaml
name: web-researcher
description: Expert at internet research and information synthesis.
x-kosmos:
  context_access: parent_summary
  builtin_tools:
    - bing_web_search
    - google_web_search
    - fetch_web_content
    - read_html_by_url
  inherit_mcp_servers: true
  inherit_skills: false
```

**System Prompt key points**:
- Define search strategy (breadth-first → depth-first)
- Require source citations
- Specify output structure (summary + key findings + source list)

---

### 9.3 Task Executor

**Characteristics**: Executes system commands and automation tasks.

**Use case**: Build and deployment, script execution, environment configuration

**Typical configuration**:
```yaml
name: build-runner
description: Executes build and deployment commands safely.
x-kosmos:
  context_access: isolated
  workspace: "D:/projects/my-app"
  builtin_tools:
    - execute_command
    - read_file
    - write_file
  inherit_mcp_servers: false
  inherit_skills: false
```

**System Prompt key points**:
- Emphasize safety (do not execute dangerous commands)
- Define allowed scope of operations
- Require confirmation before critical operations

---

### 9.4 Collaborative

**Characteristics**: Requires full understanding of conversation context.

**Use case**: Context-dependent follow-up tasks, cross-topic integration

**Typical configuration**:
```yaml
name: followup-analyzer
description: Analyzes conversation context for follow-up actions.
x-kosmos:
  context_access: full_history
  inherit_mcp_servers: true
  inherit_skills: true
  inherit_knowledge_base: true
```

**System Prompt key points**:
- Explain how to leverage conversation context
- Define what information needs to be extracted from history
- Manage token budget carefully

> ⚠️ `full_history` mode consumes a large number of tokens; use with caution. Consider whether `parent_summary` can meet your needs first.

## 10. System Prompt Best Practices

A good System Prompt is key to Sub-Agent efficiency. Here are writing recommendations:

### Principle 1: Concise and Actionable

```
❌ Bad example:
"You are a universal assistant that can do anything."

✅ Good example:
"You are a code review expert. Check the code for: security vulnerabilities, performance issues, coding style violations.
For each issue, output: file path, line number, severity (high/medium/low), fix suggestion."
```

### Principle 2: Include Clear Workflow

```markdown
## Workflow

1. Receive task description, confirm understanding
2. Use search_files to locate relevant files
3. Use read_file to review each one
4. Organize findings, sorted by severity
5. Output structured report
```

### Principle 3: Define Completion Criteria

```markdown
## Completion Criteria

Task is complete when:
- All specified files have been reviewed
- Each issue has a clear fix suggestion
- Report includes an issue statistics summary
```

### Principle 4: Add Efficiency Guidance

```markdown
## Efficiency Guidelines

- Prefer batch file reading to reduce tool call count
- Use exact matching when searching to avoid broad searches
- Stop exploring once sufficient information is found
- Do not repeat already completed steps
```

### Principle 5: Specify Output Format

```markdown
## Output Format

Output the report in Markdown format:
- First line is a one-sentence summary
- Use a table to list the issue checklist
- Provide an overall score (1-10) at the end
```

## 11. Creating Sub-Agents with AI

OpenKosmos has a built-in `sub-agent-creator` Skill that can guide AI to automatically create high-quality Sub-Agents through conversation.

### Usage

1. Ensure the Agent has the `sub-agent-creator` Skill enabled
2. Describe your needs in the conversation, for example:

   > "Help me create a Sub-Agent specialized in API documentation writing that can read code and write files."

3. The AI will guide you to clarify requirements, then generate a complete `AGENT.md` file
4. The AI will write the file to the `agents/` directory in the workspace
5. Import or confirm via Settings → Sub-Agents

### Creation Flow

The AI will guide you through these steps:

1. **Understand the purpose**: Clarify what problem the Sub-Agent should solve
2. **Determine capabilities**: Select required tools, MCP servers, Skills
3. **Design isolation**: Choose the context access mode
4. **Write System Prompt**: Generate high-quality instructions based on requirements
5. **Generate AGENT.md**: Output the complete configuration file
6. **Test and verify**: Suggested testing steps

## 12. Resource Limits and Constraints

### Runtime Limits

| Limit | Value | Description |
|------|:---:|------|
| Max parallel Sub-Agents | **5** | Upper limit of simultaneously running Sub-Agents |
| Max spawn count per session | **20** | Total number of spawns allowed within a single chat session |
| Default max turns | **25** | Upper limit of conversation turns for a single Sub-Agent |
| Context compression threshold | **60%** | Compression is triggered when 60% of the context window is reached |
| Tool result truncation | **4,000 tokens** | Excessively long tool return values are intelligently compressed by LLM |

### Hard Constraints

| Constraint | Description |
|------|------|
| **Non-recursive** | Sub-Agents cannot spawn other Sub-Agents |
| **Model inheritance** | Always uses the parent Agent's model; cannot be independently specified |
| **Non-fatal errors** | Sub-Agent execution failure does not interrupt the parent conversation; the parent Agent receives the error message and decides next steps |
| **No session persistence** | Sub-Agent conversation processes are not saved as independent session files |
| **Cancel propagation** | Canceling the parent conversation also cancels all running Sub-Agents |

## 13. FAQ

### Q: What's the difference between a Sub-Agent and using tools directly?

**A:** The main difference is that Sub-Agents have an **independent conversation loop**. They can reason over multiple turns like an independent Agent, call multiple tools in sequence, and autonomously decide next steps. Direct tool calls are single-shot executions that return a result. Sub-Agents are suited for complex subtasks requiring **multi-step reasoning**.

### Q: Why isn't my Agent delegating tasks to Sub-Agents?

**A:** Please check:
1. The Sub-Agent has been created/installed in Settings
2. It has been checked and enabled in Agent Editor - Sub-Agents Tab
3. The Sub-Agent's `description` clearly describes applicable scenarios - the AI relies on it to decide when to delegate
4. The task actually matches the Sub-Agent's expertise

### Q: Can a Sub-Agent use a different model than the parent?

**A:** No. Sub-Agents uniformly inherit the parent Agent's LLM model. This design simplifies resource management and ensures consistency.

### Q: What happens when a Sub-Agent fails?

**A:** Failure does not affect the parent conversation. The parent Agent receives the error message and decides on its own whether to retry, use a different Sub-Agent, or handle it directly. No manual intervention is needed.

### Q: Which is better, `parent_summary` or `full_history`?

**A:** Prefer `parent_summary`. It passes only about 500-800 tokens of structured summary, providing context information without wasting tokens. `full_history` passes the complete conversation history, consuming many tokens and potentially introducing irrelevant information. Use it only when the Sub-Agent truly needs to understand the details of every conversation turn.

### Q: How do I debug Sub-Agent behavior?

**A:**
1. Observe the **real-time progress card** in the chat to see which tools the Sub-Agent is calling
2. Check if the Sub-Agent's `system_prompt` is specific enough
3. Temporarily set `max_turns` to a small value (e.g., 5) to quickly observe execution patterns
4. Adjust the `description` to ensure the parent Agent delegates in the correct scenarios

### Q: How do I import Sub-Agents from Claude Code?

**A:** Place Claude Code's Sub-Agent `.md` files in the `agents/{name}/AGENT.md` directory, and OpenKosmos will automatically recognize and import them. Pure Claude Code format files without `x-kosmos` fields will have OpenKosmos extension fields populated with default values.

### Q: What's the maximum number of Sub-Agents I can create?

**A:** There is no hard limit on the number of Sub-Agents in the global registry. However, enabling too many Sub-Agents for a single Agent increases the system prompt length (each Sub-Agent injects a description block). It is recommended to enable 3-8 specialized Sub-Agents per Agent.
